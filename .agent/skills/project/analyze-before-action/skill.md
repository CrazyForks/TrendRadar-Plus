# 先分析后行动

> ⚠️ **重要**：执行前请先阅读 [common-mistakes.md](./common-mistakes.md) 避免重复错误

## 概述

当用户提出功能需求、问题修复或改进建议时，**不要立即开始编码**。应该先：

1. 理解用户的实际问题和需求
2. 分析项目中相关的现有实现
3. 评估可能的解决方案
4. 向用户提供建议并确认后再执行

## 适用场景

- ✅ 用户报告 Bug 或问题
- ✅ 用户提出新功能需求
- ✅ 用户询问某个功能是否可行
- ✅ 用户描述了一个现象但不确定原因
- ✅ 涉及多个模块的改动

## 不适用场景

- ❌ 用户明确说"帮我实现"、"开始做"、"部署"
- ❌ 简单的配置修改或文档更新
- ❌ 用户已经给出了明确的实现方案

---

## 执行流程

### 第一步：理解问题

**询问澄清**（如果需要）：
- 问题的具体表现是什么？
- 在什么场景下出现？
- 期望的行为是什么？

**不要假设**：
- 不要假设用户想要的解决方案
- 不要假设问题的根本原因

### 第二步：分析现有实现

**查找相关代码**：
```
1. 使用 grepSearch 搜索相关关键词
2. 使用 readFile 阅读相关模块
3. 理解数据流和依赖关系
```

**关键文件位置**：

| 功能模块 | 后端代码 | 前端代码 | 数据库 |
|---------|---------|---------|--------|
| 用户认证 | `kernel/auth/` | `static/js/src/auth-*.js` | `user.db` |
| 文章总结 | `kernel/user/summary_api.py` | `static/js/src/summary-modal.js` | `online.db` |
| 我的关注 | `kernel/user/preferences_api.py` | `static/js/src/my-tags.js` | `user.db` + `online.db` |
| RSS 订阅 | `kernel/scheduler/rss_scheduler.py` | - | `online.db` |
| 支付功能 | `kernel/user/payment_*.py` | `static/js/src/payment.js` | `user.db` |
| 标签系统 | `kernel/services/tag_discovery.py` | `static/js/src/article-tags.js` | `online.db` |

### 数据查询规则

**🏠 本地优先**：
- 默认查询本地数据库：`hotnews/output/online.db`
- 只有用户明确说"查询服务器"、"看下线上"时才查询远程

**本地查询命令**：
```bash
sqlite3 hotnews/output/online.db "SELECT ..."
```

**远程查询**（仅当用户要求时）：
```bash
ssh -p 52222 root@120.77.222.205 "cd ~/hotnews && sqlite3 output/online.db 'SELECT ...'"
```

### 部署规则

- 部署使用 `./deploy-fast.sh` 脚本（在 hotnews 目录下执行）
- 依赖变更时使用 `./deploy-rebuild.sh`

### 第三步：评估方案

**考虑因素**：
- 改动范围：影响哪些模块？
- 兼容性：是否影响现有功能？
- 性能：是否有性能影响？
- 数据迁移：是否需要数据库变更？

**方案对比**（如果有多个方案）：

| 方案 | 优点 | 缺点 | 工作量 |
|------|------|------|--------|
| 方案 A | ... | ... | ... |
| 方案 B | ... | ... | ... |

**影响分析**（必须包含）：

对每个方案，分析对现有功能的影响：

| 影响项 | 说明 |
|--------|------|
| ✅ 正面影响 | 改进了什么？解决了什么问题？ |
| ⚠️ 潜在风险 | 可能引入什么问题？边界情况？ |
| 🔗 关联功能 | 哪些现有功能会受影响？ |
| 📱 多端兼容 | PC/移动端/微信内置浏览器表现？ |
| 🔄 回退方案 | 如果出问题，如何快速回退？ |

### 第四步：向用户汇报

**汇报格式**：

```markdown
## 问题分析

[描述对问题的理解]

## 现有实现

[描述相关的现有代码和逻辑]

## 建议方案

### 方案 A：[方案名称]
- 实现思路：...
- 改动文件：...
- 预计工作量：...

### 方案 B（可选）：[方案名称]
- ...

## 影响分析

| 影响项 | 说明 |
|--------|------|
| ✅ 正面影响 | [改进点] |
| ⚠️ 潜在风险 | [风险点，如无则写"无"] |
| 🔗 关联功能 | [受影响的功能] |
| 📱 多端兼容 | [兼容性说明] |
| 🔄 回退方案 | [如何回退] |

## 我的建议

[推荐哪个方案，为什么]

---

请确认是否按此方案执行？或者您有其他想法？
```

### 第五步：等待确认

**必须等待用户确认后才能开始实现**：
- 用户说"可以"、"好的"、"开始"、"执行" → 开始实现
- 用户提出修改意见 → 调整方案后再次确认
- 用户说"不用了"、"算了" → 结束

## 示例对话

### 用户输入
> "我的关注标签里面有重复的标题"

### 错误做法 ❌
立即开始修改代码添加去重逻辑...

### 正确做法 ✅

**第一步：澄清问题**
> 请问是以下哪种情况？
> 1. 同一个标签卡片内出现了重复的新闻标题？
> 2. 不同标签卡片之间有相同的新闻（因为一条新闻匹配多个标签）？

**第二步：分析代码**
查看 `preferences_api.py` 中的 `get_followed_news` 函数...

**第三步：汇报**
> ## 问题分析
> 经过分析，问题是...
> 
> ## 现有实现
> 当前查询使用了 `SELECT DISTINCT`，但...
> 
> ## 建议方案
> 方案 A：在查询时按标题去重...
> 
> 请确认是否按此方案执行？

## 特殊情况处理

### 用户催促
如果用户说"快点"、"直接做"：
- 简化分析过程，但仍然给出简要方案
- 确认后再执行

### 紧急 Bug
如果是线上紧急问题：
- 快速定位问题
- 给出最小修复方案
- 可以边修复边解释

### 用户已有方案
如果用户已经给出了明确的实现方案：
- 评估方案可行性
- 如有问题指出，否则直接执行

## 检查清单

在开始编码前，确认以下问题：

- [ ] 我理解了用户的实际需求吗？
- [ ] 我找到了相关的现有代码吗？
- [ ] 我评估了改动的影响范围吗？
- [ ] 我向用户解释了方案吗？
- [ ] 用户确认了方案吗？

只有全部 ✅ 后才能开始实现。

## 实现完成后

### 第六步：确认下一步

当代码修改完成并通过语法检查后，**询问用户下一步**：

```markdown
✅ 修改完成，已通过语法检查。

改动文件：
- `path/to/file1.py`
- `path/to/file2.js`

---

还有其他需要修复或完善的吗？或者现在部署？
```

**根据用户回复**：
- 用户提出新问题 → 继续分析和修复
- 用户说"部署"、"可以了"、"没了" → 执行 `./deploy-fast.sh`
- 用户说"先不部署" → 结束，不部署

### 部署后验证

部署成功后，简要告知用户：
```markdown
✅ 部署成功！可以刷新页面验证效果。
```

### 第七步：更新项目上下文（必须）

每次完成修复或新增功能后，**必须**更新 `.kiro/steering/project-context.md` 中的"最近改动"部分：

```markdown
## 最近改动

### 2026-01-25
- ✅ [新功能/修复描述] - [涉及的主要文件]
- ✅ [另一个改动]
```

**更新规则**：
1. 每次部署成功后，添加一条记录
2. 格式：`- ✅ [简短描述] - [主要文件]`
3. 按日期分组，最新的在最上面
4. 保留最近 7 天的记录，更早的可以删除
5. 只记录功能性改动，不记录文档或配置微调

**示例**：
```markdown
### 2026-01-25
- ✅ 阅读原文不关闭弹窗 - `summary-modal.js`
- ✅ 我的关注标题去重 - `preferences_api.py`
- ✅ 总结失败追踪功能 - `summary_failure_tracker.py`, `summary_failure_api.py`
```

**注意**：这是强制要求，确保下次对话时能快速了解项目最新状态。

### 第八步：记录犯错（如有）

如果在本次任务中犯了错误（被用户指出、命令执行失败、假设错误等），**必须**更新 `common-mistakes.md`：

**触发条件**：
- 用户指出了错误
- 命令执行失败需要重试
- 假设了错误的需求或实现
- 查询了不存在的表/列
- 忘记了某个必要步骤

**记录格式**：
```markdown
### N. 错误标题
**日期**：2026-01-25
**场景**：描述发生错误的场景

**错误**：描述错误现象

**正确做法**：
- 步骤 1
- 步骤 2
```

**示例**：
```markdown
### 5. 查询数据库前未确认表结构
**日期**：2026-01-25
**场景**：查询 summary_blocked_domains 表时使用了 is_blocked 列

**错误**：实际表中没有 is_blocked 列，而是用 blocked_at 是否为 NULL 来判断

**正确做法**：
- 先用 `.schema table_name` 查看实际表结构
- 根据实际结构编写查询
```

**注意**：这是自我改进机制，确保同样的错误不会重复发生。
