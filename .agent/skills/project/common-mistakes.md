# 常见错误记录

记录 AI 在本项目中容易犯的错误，避免重复。

---

## 数据库相关

### 1. 新表未在本地创建
**问题**：添加新表的代码在 `db_online.py` 中使用 `CREATE TABLE IF NOT EXISTS`，但本地数据库是在代码添加之前创建的，导致表不存在。

**正确做法**：
- 添加新表后，需要手动在本地执行建表语句
- 或者删除本地数据库文件让其重新初始化
- 检查时先查 `.tables` 确认表是否存在

### 2. 查询不存在的列
**问题**：代码中的表结构与实际数据库不一致（如 `is_blocked` vs `blocked_at`）。

**正确做法**：
- 先用 `.schema table_name` 查看实际表结构
- 根据实际结构编写查询

## SSH 远程操作

### 1. 未经确认直接执行 SSH
**问题**：直接执行 SSH 命令可能被用户拒绝。

**正确做法**：
- 先询问用户是否允许执行
- 用户确认后再执行
- 格式：`需要查询服务器数据，是否允许执行以下命令？`

### 2. SSH 命令格式
**正确格式**：
```bash
ssh -p 52222 root@120.77.222.205 "cd ~/hotnews && sqlite3 output/online.db 'SELECT ...'"
```

## 部署相关

### 1. 忘记部署
**问题**：代码修改完成后忘记询问用户是否部署。

**正确做法**：
- 修改完成后必须询问：`还有其他需要修复或完善的吗？或者现在部署？`
- 用户确认后执行 `./deploy-fast.sh`

### 2. 部署后忘记更新 project-context
**问题**：部署成功后忘记更新 `.kiro/steering/project-context.md`。

**正确做法**：
- 部署成功后，更新"最近改动"部分
- 格式：`- ✅ [描述] - [文件]`

## 代码修改

### 1. 假设用户需求
**问题**：没有澄清就开始编码。

**正确做法**：
- 先分析问题
- 提出方案
- 等待用户确认后再实现

### 2. 忘记语法检查
**问题**：修改代码后没有检查语法错误。

**正确做法**：
- 使用 `getDiagnostics` 检查修改的文件
- 确保无错误后再告知用户

## 历史错误记录

### 1. 查询数据库前未确认表结构
**日期**：2026-01-25
**场景**：查询 summary_blocked_domains 表时使用了 is_blocked 列

**错误**：实际表中没有 is_blocked 列，而是用 blocked_at 是否为 NULL 来判断是否被屏蔽

**正确做法**：
- 先用 `.schema table_name` 查看实际表结构
- 根据实际结构编写查询

### 2. 未确认本地数据库表是否存在
**日期**：2026-01-25
**场景**：检查本地 summary_failures 表数据时

**错误**：直接查询表，但本地数据库中表不存在（代码是后加的，本地数据库未重新初始化）

**正确做法**：
- 先用 `.tables` 确认表是否存在
- 如果不存在，手动执行建表语句或提示用户重新初始化数据库

### 3. 默认查询服务器而非本地
**日期**：2026-01-25
**场景**：用户问"系统有没记录到失败"，我直接询问是否查询服务器

**错误**：应该先查询本地数据库，只有用户明确要求时才查询服务器

**正确做法**：
- 默认查询本地：`sqlite3 hotnews/output/online.db "SELECT ..."`
- 只有用户说"查询服务器"、"看下线上"时才查询远程

### 4. 查询时使用不存在的列名
**日期**：2026-01-25
**场景**：查询 summary_failures 表时使用了 `failure_reason`、`failure_count` 等列名

**错误**：实际表中列名是 `reason`、`attempt_count`，不是 `failure_reason`、`failure_count`

**正确做法**：
- 先用 `.schema table_name` 查看实际表结构
- 使用实际存在的列名：`reason`, `attempt_count`, `first_failed_at`, `last_failed_at`
- 不要凭记忆或猜测列名
