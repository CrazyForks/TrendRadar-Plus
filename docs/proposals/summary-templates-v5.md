# AI 总结深度优化 V5

## 背景

V4 版本实现了：
- 10 种文章类型 + 1 个通用兜底
- 模板结构趋同，仅尾部差异化
- 置信度降级机制
- 来源预分类提示

**用户反馈**：总结过于简单，缺乏深度分析。用户希望看到更有洞察力的内容，而不仅仅是信息提取。

---

## 核心设计原则

### 原则 1：按类型差异化优化

不是所有类型都需要深度分析，应该根据类型特点针对性优化：

| 类型分组 | 包含类型 | 优化方向 | 理由 |
|----------|----------|----------|------|
| **深度分析型** | news, policy, business, opinion, research | 增加深度分析、批判性视角 | 这些内容用户需要理解"为什么"和"意味着什么" |
| **技术专业型** | tech (新增) | 技术原理解读、技术栈分析、上手指南 | AI/开发类内容占比 37%，需要专门优化 |
| **实用导向型** | tutorial, product, listicle | 强化行动建议、资源链接 | 用户关心"怎么用"和"去哪找" |
| **轻量信息型** | lifestyle, interview, general | 保持简洁，适度优化 | 用户只需快速了解要点 |

### 原则 2：根据原文长度动态调整

原文长度不同，总结深度也应不同：

| 原文长度 | 总结策略 | 输出字数 |
|----------|----------|----------|
| **短文** (<1000字) | 精简模式：只保留核心要点，不强求深度分析 | 150-300 字 |
| **中文** (1000-3000字) | 标准模式：完整模板，适度分析 | 300-500 字 |
| **长文** (3000-8000字) | 深度模式：完整模板 + 深度分析模块 | 500-800 字 |
| **超长文** (>8000字) | 精华模式：提炼核心 + 章节概要 | 600-1000 字 |

### 原则 3：渐进式优化

先优化最有价值的类型，验证效果后再扩展：
1. **Phase 1**：优化 news、business、opinion（用户最常阅读）
2. **Phase 2**：优化 policy、research（专业用户需求）
3. **Phase 3**：优化其他类型（根据反馈决定）

---

## 各类型优化方案

### 一、深度分析型（news, policy, business, opinion, research）

这类文章用户需要理解背景、影响和可信度，是优化重点。

#### � 新闻资讯（news）

**当前问题**：
- 只提取事实，缺少"为什么重要"
- "后续关注"过于泛化

**优化方案**：

```markdown
## 🎯 核心事件
（一句话概括：谁+做了什么+结果）

## 💡 关键信息
- 要点1
- 要点2
- 要点3

## 🔍 为什么重要
（2-3句话：这件事的意义、与更大背景的关联、可能的连锁反应）

## 📌 后续关注
- 具体的时间节点或信号（不是泛泛的"持续关注"）
```

**长度策略**：
- 短新闻(<1000字)：省略"为什么重要"，直接给结论
- 中等新闻：标准模板
- 长新闻(>3000字)：增加"各方反应"或"历史脉络"

---

#### 📊 商业分析（business）

**当前问题**：
- 数据罗列，缺少商业洞察
- 没有帮助用户判断信息质量

**优化方案**：

```markdown
## 🎯 核心结论
（一句话：这篇分析的核心观点）

## 💡 关键数据
- 数据1 + 同比/环比
- 数据2 + 行业对比
- 数据3

## 🔍 商业洞察
- **背后逻辑**：为什么会这样，商业模式/战略意图
- **行业影响**：对竞争格局、上下游的影响
- **信号判断**：这是个案还是趋势

## ⚖️ 需要注意
- 数据来源/可信度
- 文章可能的立场或偏见
- 未提及的风险因素

## 📊 关注信号
- 下一个关键时间节点
- 需要跟踪的指标
```

**长度策略**：
- 短文：省略"需要注意"
- 长文(>5000字)：增加"分章节要点"

---

#### 💭 观点评论（opinion）

**当前问题**：
- 只提取论点论据，缺少批判性评估
- "延伸思考"过于空洞

**优化方案**：

```markdown
## 🎯 核心观点
（一句话：作者认为 XXX，因为 YYY）

## 💡 主要论据
- 论据1
- 论据2
- 论据3

## ⚖️ 批判性视角
- **作者背景**：可能的立场或利益关联
- **论证质量**：论据是否充分，逻辑是否严密
- **另一面**：反对者可能怎么说

## 💭 延伸思考
- 一个具体的思考问题（不是泛泛的"值得思考"）
```

**长度策略**：
- 短文：省略"批判性视角"，保留核心观点
- 长文：可增加"观点演变"或"相关争议"

---

#### ⚠️ 政策解读（policy）

**优化方案**：

```markdown
## 🎯 政策核心
（一句话：这个政策是关于什么的）

## 💡 关键变化
- 变化1：之前 vs 现在
- 变化2
- 变化3

## 🔍 深度解读
- **出台背景**：为什么现在出这个政策
- **真实意图**：政策制定者想解决什么问题

## ⚠️ 影响评估
- **直接影响**：谁受影响，怎么影响
- **间接影响**：可能的连锁反应
- **时间窗口**：什么时候生效，有多少准备时间

## 🎯 应对建议
- 针对不同群体的具体建议
```

---

#### � 研究报告（research）

**优化方案**：

```markdown
## 🎯 核心发现
（一句话：这篇研究发现/证明了什么）

## 💡 关键结论
- 结论1 + 数据支撑
- 结论2
- 结论3

## 🔍 研究评估
- **方法可靠性**：样本量、研究方法是否合理
- **适用范围**：结论在什么条件下成立
- **局限性**：研究的不足之处

## � 实践意义
- 这个发现对实际工作/生活有什么启示
```

### 二、技术专业型（tech - 新增）

AI/开发类内容占比 37%，需要专门的模板来处理技术深度分析。

#### 🤖 技术解读（tech）

**覆盖场景**：
- AI 新产品发布（如 GPT-5、Claude 4）
- AI 技术解读（如 RAG 原理、Agent 架构）
- 开发工具推荐
- 编程教程
- AI 行业分析
- 开源项目介绍

**优化方案**：

```markdown
## 🎯 核心内容
（一句话：这篇文章讲什么技术/工具/产品）

## 💡 技术要点
- 要点1：技术原理或核心功能
- 要点2：解决什么问题
- 要点3：与现有方案的对比

## 🔍 技术分析
- **核心价值**：这个技术/工具解决什么痛点
- **技术栈**：涉及的语言/框架/依赖
- **适用场景**：什么情况下适合使用
- **局限性**：已知的限制或不足

## 🛠️ 快速上手
- **官方资源**：[文档/GitHub](链接)
- **前置要求**：需要什么基础
- **第一步**：最简单的尝试方式
```

**长度策略**：
- 短文(<1000字)：省略"技术分析"，保留核心要点
- 长文(>3000字)：增加"技术细节"或"代码示例说明"

## 动态长度控制

### 实现方案

在 prompt 中根据原文长度动态调整指令：

```python
def get_length_instruction(content_length: int, article_type: str) -> str:
    """根据原文长度返回长度控制指令"""
    
    # 深度分析型有更大的弹性空间
    is_deep_type = article_type in ['news', 'policy', 'business', 'opinion', 'research']
    
    if content_length < 1000:
        return """
【长度控制】原文较短，请精简输出：
- 省略深度分析模块
- 总结控制在 150-300 字
- 只保留最核心的信息
"""
    elif content_length < 3000:
        return """
【长度控制】标准模式：
- 完整输出所有模块
- 总结控制在 300-500 字
"""
    elif content_length < 8000:
        if is_deep_type:
            return """
【长度控制】原文较长，请深度分析：
- 完整输出所有模块，深度分析部分可以展开
- 总结控制在 500-700 字
- 重点挖掘"为什么"和"意味着什么"
"""
        else:
            return """
【长度控制】原文较长：
- 完整输出所有模块
- 总结控制在 400-600 字
"""
    else:
        return """
【长度控制】原文很长，请提炼精华：
- 完整输出所有模块
- 可增加"章节概要"帮助理解结构
- 总结控制在 600-900 字
- 重点是提炼核心，不是面面俱到
"""
```

### Prompt 模板调整

在 user prompt 末尾动态插入长度指令：

```python
user_prompt = f"""
{template_content}

{length_instruction}

---

{content}
"""
```

## System Prompt 优化

针对深度分析型文章，优化 system prompt：

### 📰 新闻资讯

```python
# V4
"""你是新闻编辑，用客观简洁的语言提炼新闻要点。"""

# V5
"""你是资深新闻分析师。你的任务不只是提炼事实，更要帮助读者理解这件事为什么重要。

分析要求：
1. 先给结论：一句话说清楚发生了什么
2. 挖掘意义：这件事为什么值得关注，与更大的背景有什么关联
3. 具体建议：后续关注什么，要给出具体的时间节点或信号，不要说"持续关注"

避免：
- 空洞套话（如"意义重大"、"值得关注"）
- 没有依据的推测
"""
```

### 📊 商业分析

```python
# V5
"""你是商业分析师，帮助读者理解商业事件背后的逻辑和影响。

分析要求：
1. 核心结论：这篇分析最重要的观点是什么
2. 数据解读：不只是罗列数字，要解释数字意味着什么
3. 商业洞察：背后的商业逻辑、战略意图、行业影响
4. 风险提示：数据来源是否可靠，有什么被忽略的风险

避免：
- 只罗列数据不解读
- 对所有公司都说"前景看好"
"""
```

### 💭 观点评论

```python

2. 评估论证：论据是否充分，逻辑是否严密
3. 识别立场：作者可能的背景和利益关联
4. 补充视角：反对者可能怎么说，有什么没被讨论
```

## 成本影响评估

| 场景 | V4 | V5 预估 | 变化 |
|------|-----|---------|------|
| 短文(<1000字) | ~300字 | ~200字 | -33% |
| 中等文章 | ~350字 | ~400字 | +14% |
| 长文(>3000字) | ~400字 | ~600字 | +50% |
| 平均 token 消耗 | ~2000 | ~2200 | +10% |

**结论**：通过动态长度控制，短文反而更精简，长文才增加深度，整体成本增加可控。

- ✅ 短文不会被强行拉长，长文能充分展开
